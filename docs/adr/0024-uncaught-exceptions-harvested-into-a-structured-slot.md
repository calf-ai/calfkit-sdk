# Uncaught exceptions are harvested into a structured `ErrorReport.exception` slot, not flattened to a string

A fault synthesized from an arbitrary uncaught exception (`ErrorReport.from_exception`) now carries a typed `ExceptionInfo` slot — `type`, `module`, and a JSON-native `attrs` dict harvested from the exception's `vars()` — and the Python exception cause chain (`__cause__`-only) is mapped onto the existing recursive `causes` field, one lean `ErrorReport` per link, each with its own `exception` slot. Previously the exception was flattened to `message = str(exc)` plus a `details["calf.exception_type"]` class-name breadcrumb, discarding all structured state (a provider's `status_code`, `body`, error `code`) and the entire cause chain. We did this because the structured information is the diagnostic value — a consumer should read `report.exception.attrs["status_code"]`, not regex a message string — and because the data round-trips and is read at the client with no producer class importable, keeping producer and consumer deployments version-independent.

This **extends, and does not break, ADR-0006's principle that exception identity never crosses the wire**: we ship the exception's *data*, never the object. The catch-all `error_type` is renamed `calf.unhandled` → `calf.exception` to name the origin and pair with the new slot; the redundant `calf.exception_type` breadcrumb is removed (subsumed by `exception.type`).

## Considered options

- **Serialize the exception object and rehydrate it on the receiver** (native binary object serialization, or hand-written per-class pydantic schemas). Rejected: native object serialization executes arbitrary code on decode — unacceptable across a shared message bus / trust boundary — and does not even round-trip the exceptions we care about (verified: `ModelHTTPError`'s default reconstruction recipe replays `self.args` and rebuilding raises `TypeError`). Both approaches also require the producer's exception class importable on the receiver at a compatible version, re-coupling deployments and reversing ADR-0006. We ship data, not objects.
- **A separate `ExceptionInfo.cause` chain** distinct from `causes`. Rejected: faithful, but duplicates the bounding + cycle-guard machinery and forces a consumer to traverse two places. Reusing `causes` gives one traversal (`walk()`/`find()`) and the existing depth/total budgets and cycle guard for free.
- **Curated `error_type` classification in the same change** (detect a context-window `ModelHTTPError` → mint `calf.model.context_window_exceeded`). Deferred: the slot is permanent infrastructure; classification is the Option A successor (ideally at the LiteLLM boundary, #230) that rides this same slot later. Bundling would conflate "carry the data" with "name the failure," and `FaultTypes.MODEL_CONTEXT_WINDOW_EXCEEDED` stays unproduced for now.

## Consequences

- **No leak prevention.** The harvest is unconditional — exceptions are treated as non-leaking, since calfkit user payloads live on frame inputs / `message_history`, never as exception attributes. An escaped exception's attributes are protocol/runtime metadata.
- **Totality is load-bearing.** `from_exception` is called unguarded at the rail chokepoint (`base.py:669`, and a second direct call at `:1070`), so the harvest must never raise — and because it runs *before*/as input to `build_safe`, `build_safe`'s own fallback cannot cover it. The harvest is therefore made fully total in itself: it guards `vars()`, the `.items()` iteration, and the entire per-attr body (the key compare, the `str(k)` coercion, and the serialize) with bare `except`, guards the type-access and `ExceptionInfo(...)` construction, and `_walk_exception_chain` guards the `__cause__` read; the serializer is hardened (`serialize_unknown` + `bytes_mode='base64'` + `inf_nan_mode='strings'`, the existing dependency, no new one), and the residual deep-nesting drop (plus a `__cause__`-chain deeper than the cap) is tallied under `details['calf.elided']`, never silent.
- **`__context__` was dropped (`__cause__`-only).** Implicit `__context__` chaining was evaluated and rejected: `from_exception` runs inside the rail's `except` blocks, where Python auto-sets `__context__` to the framework's caught exception — walking it would leak internals onto the wire (the rail's `_SeamAccidentError` wrapper) and duplicate the recovery `cause`. `__cause__` (explicit `raise ... from`) is the clean, intentional lineage and is all the target case needs.
- **`ExceptionGroup` branching is deferred.** The walk is linear (`__cause__` only); an `ExceptionGroup`'s `.exceptions` is a getset descriptor, **not in `vars()`**, so a `FallbackExceptionGroup`'s per-model failures are **dropped entirely** (not even string-coerced), pending the deferred branching follow-up. The same applies to any exception-valued plain attribute and the implicit `__context__` chain.
- **`NodeFaultError` mints are unchanged.** Deliberate mints convert verbatim (the mint rule) and carry no `exception` slot — they already supply a complete report; there is no foreign exception to introspect.
- **Hard break (pre-1.0):** the `calf.unhandled` wire value and the `calf.exception_type` details key both retire. Full design in `docs/designs/exception-harvest-spec.md`.
