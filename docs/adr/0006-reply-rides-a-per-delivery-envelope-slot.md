# Replies ride a per-delivery envelope slot; correlation is transport-echoed

Every reply (success or fault) crosses the wire in a new per-delivery `Envelope.reply` slot — `ReturnMessage(parts)` XOR `FaultMessage(error)` — classified by a framework-stamped `x-calf-kind: call | return | fault` header and correlated to its pending call by transport echo (`in_reply_to` = the popped frame's id; `tag` = the caller's opaque token, e.g. `tool_call_id`, written on the `CallFrame` at `Call` time). Content is the existing `ContentPart` vocabulary, typed at the read site (`output_type` projection); requests stay typed at the write site (callee-declared `@handler` schemas). `State` is demoted to an opaque traveling record: the blob-write result protocol (`state.tool_results` carriage, `final_output_parts`) is deleted.

We chose this because requests and replies have asymmetric lifetimes and interpreters, and the carriage should match both: a call's input lives on the durable frame (activation-record pattern — it must survive the callee's whole multi-delivery invocation), while a reply exists for exactly one hop and its frame is popped by the very publish that carries it (JSON-RPC's response-references-request-by-`id` shape); a request has one specific interpreter (the callee → schema-on-write) while a reply has many generic ones — the calling agent's model materializer, catch-all seams, sinks, the client — none of which can know the callee's concrete type over topic addressing (→ schema-on-read over a renderable part vocabulary; MCP's schema'd-arguments-in / content-blocks-out split is the closest precedent). The tag echo makes any `ReturnCall`-ing node a protocol-correct callee with zero caller-side correlation maps — agent-as-callee composes without adapters.

## Considered options

- **Results written into the traveling `State` by convention** (the shipped design: tool writes `tool_results[id]`, agent writes `final_output_parts`, caller digs them out). Rejected: makes the whole session record the inter-node interface — every consumer needs a shovel (`project_output`, sibling-state scans, POV projection workarounds), only blob-protocol-speaking nodes can answer a call slot, and stale single-overwrite slots leak previous turns' outputs to observers.
- **Fault carriage via frame-payload reuse + reserved `x-calf-route` values** (`calf.return`/`calf.fault` — the first spec draft). Superseded: required four patch rules (dead-payload frame overwrite, route-header overloading against its documented ingress-only contract, `origin_payload` as content-borne correlation, a load-bearing never-match-`'*'` rule). Needed patch rules = wrong factoring.
- **A fourth `event` kind for broadcast mirrors.** Rejected: on a broadcast medium any consumer can tap any topic, so a header can state facts about the *message* only, never a receiver's relationship to it; observer-vs-actor is the node type's property (the consumer role), and with that placed correctly the extra kind does no work.
- **Transport-level kind multiplexing (a per-node error topic; topic-per-kind lanes).** Rejected for the point-to-point rails: topics multiplex transport properties (audience/ACL, QoS class, ordering-coupling); headers multiplex semantics within one audience. Returns and faults share every transport property, so splitting them forfeits the lock-free per-workflow serialization one keyed topic provides while the kind header survives regardless (the client reply topic and broadcast mirrors are mixed-kind). The *opposite* collapse — a **single entrypoint topic** per node for all kinds — was also explored and rejected (2026-06-12): it deleted co-location machinery but forbade shared-ingress listening (a node's entrypoint would double as its reply address, so co-listening nodes would consume each other's continuations — an identity-blind workflow fork) and demoted two by-construction guarantees (return-topic uniqueness derived from node id; framework-owned topic config) to silently-violable deployment conventions. The locality problem is resolved by the durable fan-out aggregator instead (spec §7.7). The ops wants (fault firehose, long fault retention) belong to the deferred fault-stream/DLT topic on the observability plane.
- **Write-site response schemas per node (gRPC-style symmetry).** Rejected: catch-all interposers go blind on opaque typed blobs; the model boundary needs a universal renderable vocabulary anyway (else per-callee renderers); the static guarantee is unredeemable over topic strings.

## Consequences

- `x-calf-kind` ⇔ reply-shape agreement is a construction invariant of the single publish chokepoint (`_publish_action` stamps both), not a rejection rule — receivers tolerate disagreement as stray, never fault on it.
- `CallFrame` additionally carries a durable fan-out marker (`fanout_id`, stamped on the caller's frame in each sibling stack copy) so a fan-out reply arriving after its in-process batch was lost is statelessly detectable as an orphan and floored — without it, lost-batch siblings would body-dispatch as live continuations and fork the workflow.
- `ModelRetry` renders to self-contained text at origin, carried as an ordinary `TextPart` with a `calf.retry` metadata marker (no new part kind): the marker exists because the providers are asymmetric — Anthropic maps retry prompts to `tool_result(is_error=True)` vs `is_error=False` for returns — and the agent's materializer honors it to preserve that fidelity; it doubles as the retry-counting hook for the successor retry issue.
- `tool_call_id` appears twice on a tool-call message (in `ToolCallRef` as tool input; as the frame `tag` for correlation) — same datum, two roles, two layers; accepted over callee-side correlation copying.
- The conversion layer collapses: one value→parts coercion, one parts→value projection, and the agent's slot materialization as the SDK's only per-type codec; consumers and custom nodes need none.
- Hard break (pre-1.0): every wire artifact changes shape; `FailedToolCall`, `final_output_parts`, and the `CalfToolResult` wire union all retire with it.
