## 1. Summary

An agent can **discover other agents at runtime** (no imports, no object references) and interact with them two ways:

- **Message** a peer — request/reply consultation via the **`message_agent(name, message)`** tool (a **Peer message**): the peer is consulted on a fresh sub-conversation and its reply folds into the calling tool call's result; the caller keeps control.
- **Hand off** to a peer — transfer of conversation control by producing a **`HandoffRequest(name, message)`** as the turn's output (a structured-output option, *not* a tool): the calling agent ends its turn, the peer continues the *full* conversation and returns to the original caller; the handing agent drops out.

Both are configured by **one ctor param, `peers=`**, holding **capability-typed handles** — `Messaging(...)` and `Handoff(...)` — each scoped curated (named) or open (`discover=True`), independently per capability. Each capability is surfaced to the model **only if** its handle is present — messaging as a built-in tool, handoff as an output option — with a directory rendered **fresh every turn** from a live `calf.agents` view. Both ride the same discovery substrate; they differ in the control-flow primitive (a `Call` that returns vs. a `TailCall` that transfers).

## 2. Goals / Non-goals

### Goals
1. A base-node **private input topic** (`{node_kind}.{name}.private.input`), auto-subscribed on every node — a deterministic `name → topic` map (§4.1).
2. `AgentCard` on a compacted `calf.agents` topic, advertised by every agent (§4.2).
3. A unified consume surface — `peers=[Messaging(...), Handoff(...)]` — with per-capability curated-XOR-open scope and per-capability tool gating (§5.1).
4. **Messaging:** the `message_agent` external tool + caller-side Peer-message adapter (§5.2).
5. **Handoff:** a `HandoffRequest` **structured-output union member** that emits a `TailCall` transferring control (§5.3).
6. A **cycle guard** for messaging recursion (§8).

### Non-goals (deferred)
- No `find_agents` search tool (the per-turn directory is the surface); a finder is the large-fleet escape hatch (§14).
- **No callee-side control for either capability** — caller-side scope is the only v1 trust boundary; visibility + admission deferred whole to #231.
- **No handoff-loop guard** — handoff loops are knowingly unbounded in v1 and out of reach of the messaging cycle guard (§8); #251 is the eventual backstop.
- No resource budgets / per-call deadline / dangling recovery — #251.
- No Enlist mode (shared live conversation).
- No general "call any node by name" caller surface — the §4.1 primitive is built now; that broader feature reuses it later.

## 3. Shared substrate

Both capabilities resolve a peer **by name** against the same machinery:
- **Discovery:** the `calf.agents` view (§4.2) — a per-turn directory of live peers.
- **Addressing:** `derive_input_topic(name) = agent.{name}.private.input` (§4.1) — no topic is advertised; the caller derives it from the name.
- **Identity for the cycle guard:** per-frame `caller_node_id` (§8).

## 4. The advertise side

### 4.1 Base-node addressing primitive (ADR-0017)

Every `BaseNodeDef` exposes a deterministic **private input topic**, and **every node subscribes to it**:

```python
@property
def _private_input_topic(self) -> str:
    return f"{self._node_kind}.{self.name}.private.input"   # agent.planner.private.input, tool.add.private.input, ...
```

- **Contributed at registration like `_return_topic`** (a property read in `register_handlers` and `topics_for_nodes`, **flagged `framework=True`**) — *not* appended into `subscribe_topics` in `__init__` (the `@dataclass` tool/MCP node `__init__`s bypass `BaseNodeDef.__init__`, so a list-append would miss them). `_node_kind` is a `ClassVar`, available before instance init.
- **Why universal (every node):** a deterministic `name → topic` map for all kinds, reused later by a general call-by-name caller surface. *(Conditional further payoff: **if** tool public inboxes are later standardized onto the derived topic, dispatch topics become derivable and the capability plane could drop `dispatch_topic` — not automatic, since a tool's `dispatch_topic` is today its arbitrary `subscribe_topics[0]`.)* For non-agents in v1 this inbox is **dormant** (nothing dispatches to it — `derive_input_topic` hard-codes `agent.`); one provisioned topic per node, no traffic.

### 4.2 `AgentCard`

```python
AGENT_CARD_SCHEMA_VERSION = 1
AGENT_CARD_DESCRIPTION_MAX = 512   # chars — a short directory blurb, not a system prompt

class AgentCard(ControlPlaneRecord):     # topic: calf.agents
    schema_version: int = AGENT_CARD_SCHEMA_VERSION
    description: str | None = Field(default=None, max_length=AGENT_CARD_DESCRIPTION_MAX)
    # identity is the wire KEY (the agent's name); liveness + node_kind worker-stamped.
    # No input topic (derived from name, §4.1), no system prompt, no tool list.
```

- Card = `name` (key) + optional `description` + inherited liveness. `node_kind` is worker-stamped (`"agent"`); informational on this single-record-type topic (contract: only agents advertise here, so `derive_input_topic` may hard-code `agent.`).
- **`description` bounded loudly** — `Field(max_length=…)` makes an over-long value a construction error (never silent truncation; never an oversize record that decode-fails and silently drops the node).
- **`@advertises(topic="calf.agents", record=AgentCard)` is new code** on `BaseAgentNodeDef`; every agent advertises (no opt-out — §7); the worker-owned publisher heartbeats/tombstones it.

## 5. The consume side

### 5.1 Surface — `peers=[Messaging(...), Handoff(...)]`

A list of **capability-typed handles**, each owning its own curated-XOR-open scope (mirroring the shipped `Tools` handle):

```python
@dataclass(frozen=True)
class Messaging:   # feeds message_agent
    names: tuple[str, ...]
    discover: bool = False
    # CUSTOM __init__ for varargs (a vanilla @dataclass can't bind *names — mirrors the shipped Tools, tool.py):
    #   def __init__(self, *names: str, discover: bool = False): coerce Sequence[str] -> frozen tuple; dedupe;
    #   validate neither/both -> ValueError, empty-string name -> ValueError.
    # NOTE (M2): the agent's-own-name reject is NOT here — a handle doesn't know the
    # enclosing agent's name; that check lives in the Agent ctor where peers= is processed.

class Handoff:     # feeds handoff — identical shape (custom __init__, same validation)
    ...
```

```python
Agent(name="triage", peers=[
    Messaging("billing", "support"),    # may message these
    Handoff("refunds"),                 # may hand off to this
    Messaging(discover=True),           # (alt) may message any live agent
])
```

- **Capability = the type** (no boolean flags); each handle owns curated names *xor* `discover=True`. **Scopes are independent per capability** — `Messaging(discover=True)` opens messaging only; handoff scope is governed solely by `Handoff` handles, and vice versa.
- **Within a capability: curated XOR open** — a `discover=True` handle is the exclusive author of that capability's scope (no named handle of the same kind alongside it), the same rule the **shipped** discover mode (`Tools(discover=True)`, #270; ADR-0014 `accepted`) enforces; `Messaging`/`Handoff` mirror that shipped precedent. Multiple same-kind handles are kept **independent** and resolved separately each turn (the shipped `Tools` has no `merge`); the rendered directory dedupes by name. An empty handle (no names, `discover=False`) is a construction error.
- **Per-capability gating:** the model is given the `message_agent` **tool iff** a `Messaging` handle is present, and `HandoffRequest` is added to its **`output_type` iff** a `Handoff` handle is present **and ≥1 in-scope peer is live that turn** (an empty live set omits the member — a `Literal` has no empty form, §5.3). No handle → no surface. (Two-level: handle presence gates whether the capability *exists*; per-turn liveness gates whether it is *offered* and to whom — messaging always renders the tool, with an empty directory as a "none reachable" body; handoff omits the member on an empty live set and conveys the dormant capability via an ephemeral instruction instead, §5.3.)
- **Constructor-only** (closes the read-side-wiring timing hole). `peers=` is distinct from `tools=` (tool nodes/MCP); no interaction. `peers=[]` ≡ `peers=None` — no handles → no a2a surfaces and no read-side view.
- **No cross-absorption with `tools=` (M4).** `peers=` **type-validates** each element is exactly `Messaging`/`Handoff` (anything else — incl. a `Tools`/`MCPToolbox`/`ToolBinding` — raises `TypeError`). Conversely, `Messaging`/`Handoff` deliberately do **not** implement the tool protocols (`resolve_tools`/`tool_bindings`), so a handle mistakenly placed in `tools=` falls through `split_tool_declarations`' `isinstance` routing to its existing `else → TypeError` rather than being silently absorbed (`ToolSelector`/`ToolProvider` are `@runtime_checkable`, matched structurally by method name).
- Self is rejected in the **Agent ctor** where `peers=` is processed (own name in a handle's `names` — the handle itself can't check this, it doesn't know the agent's name; M2), at dispatch (§5.2/§5.3), and excluded from the render.

### 5.2 Messaging — `message_agent` (external tool)

A **Peer message** is request/reply, modeled as a tool call; the caller keeps control.

- **Tool & directory:** `message_agent(name, message)` — a real member of the model's `ExternalToolset` (injected at the `ExternalToolset` build alongside `tools_registry`; the dispatch sites fork on its well-known name **before** the `tools_registry` lookup, so it's orthogonal to `override_agent_tools`). Its description renders the **Messaging-scoped live** directory per turn (`name — description`, sorted, self-excluded; empty → "none reachable" body) — **new render code, not a `Tools` copy** (`Tools` produces flat tools with no directory). A curated name absent from the live view at render emits a **one-time WARNING** (mirroring the unresolved-selector warning in selector resolution) and is omitted; targeting it → `RetryPromptPart`. The `message_agent` name is **reserved against a construction-time tool** (an eager or named-`Tools` user tool named `message_agent` → construction error); because the built-in is injected into the `ExternalToolset` *outside* `tools_registry`, this is checked against the construction-time surface (eager `self.tools` + named `Tools` selectors) that the intra-`tools_registry` discover-merge collision guard would not see. *(Reserving the name against a **runtime discover-resolved** tool node of the same name — the narrow case where the appended built-in would silently shadow the discovered node — is **deferred to a follow-up**; PR-B ships the construction-time reservation only.)*
- **What the peer receives:** a *fresh* sub-conversation (only the `message`), with `deps`/`correlation_id` carried forward. The seeded `message` lands as a **user-role turn** the peer acts on — untrusted input (prompt-injection surface; in open mode any agent can inject one); the peer must guard it.
- **No callee handler** — a peer invoked over the standard path runs its existing `@handler('*')` and replies through the reply slot. Slot routing is generic (`_resolve_callee`, `base.py:992`); peer faults run the caller's `on_callee_error` for free. **Result rendering** is the one bit: a peer's multi-part reply (`extract_lenient` would drop the preamble) is **serialized whole** — Text verbatim, Data JSON-encoded, File as `[file: <media_type> <uri>]`, newline-joined, empty → `(no content)` — discriminated at fold time via `state.tool_calls[tag].tool_name == message_agent` (`None`-guarded).
- **Dispatch (§5.4) → `Call`** with a fresh seeded sub-state, `callback`=the caller's return slot, `tag`=tool_call_id. **A `message_agent` call always rides the durable fan-out batch (snapshot + restore), even when it is the turn's only call** — it must *not* take the single-`Call` fast path (C1). That fast path adopts the *callee's* returned state, which is correct only for a callee that round-trips the caller's **own** state (a tool — it never authors `message_history`); a peer runs on a **fresh seeded** state and authors its own sub-conversation, so without a snapshot the caller would resume on the *peer's* history. The signal that forces the batch is an explicit **`isolate_state=True`** on the `message_agent` `Call` (§5.4/L13) — base.py opens a durable batch for any call so marked (or for N≥2), so a lone *tool* keeps the fast path. The durable batch snapshots the caller's state at dispatch and restores it at fold, so the reply folds into `tool_results[tag]` onto the caller's own conversation (§5.4). **Bad target** (unknown/offline/out-of-scope/self/cycle) → model-visible `RetryPromptPart`, no dispatch.

### 5.3 Handoff — `HandoffRequest` (structured-output union member)

A **Handoff** transfers conversation control; the handing agent relinquishes. The model expresses it by producing a **`HandoffRequest`** as its turn's output — **not** by calling a tool.

- **A structured-output union member, not an output tool.** When a `Handoff` handle is present **and ≥1 in-scope peer is live that turn** (an empty live set omits the member — see the empty-set bullet below), `HandoffRequest` is added to the agent's `output_type` as a **bare** model: `output_type = [final_output_type, HandoffRequest, DeferredToolRequests]`. The turn ends with **either** the normal answer **or** a `HandoffRequest`. pydantic-ai builds a discriminated union (a `kind`-tagged `anyOf`, `_output.py`), so the choice is explicit; calfkit discriminates by **type** (`isinstance(result.output, HandoffRequest)`) for *dispatch*. (The cross-agent *rendering* of the union-renamed output tool `final_result_<Type>` is handled by the `_surface` `final_result`-prefix match (**landed, #276**) — see the handoff turn below; together these close the former C1b.)
- **Terminality + exclusivity are mode-dependent (not "by construction").** Producing the output ends the run, so a handoff is **terminal**. **Exclusivity** — a handoff not being executed alongside a sibling tool call — is **not** purely structural and depends on the resolved output mode: in **tool** mode the `HandoffRequest` is a `final_result_*` output-tool call, and a co-emitted function/external tool call is dropped only because calfkit runs `end_strategy='early'` (the vendored default, never overridden — it stubs siblings once a final result is set); under **native/prompted** mode (the thinking path) the `HandoffRequest` is JSON *text*, and pydantic-ai **prioritizes tool calls over text** (it processes any co-emitted tool call and **defers** the text), so exclusivity there rests on the model *not* co-emitting a tool call with the handoff. Both hold in practice (early + models rarely mix a tool call with a final structured output), but neither is a construction guarantee — a co-emitted tool call **defers** (never corrupts) the handoff (§9).
- **Why a *bare* model (the regime fix — resolves the former C1).** The safety hinges on `allow_text_output`: a `ToolOutput` wrapper forces the schema into `tool` mode with `allow_text_output=False`, which Anthropic rejects with extended thinking (it would force `tool_choice=required`, incompatible with thinking — `anthropic.py`). Two **bare** union members keep `allow_text_output=True`, so that guard never fires — via a mode that depends on `final_output_type`: a **structured** type + bare `HandoffRequest` builds an `AutoOutputSchema` (`auto` mode, *identical to a no-handoff structured agent* — degrades to native/prompted under thinking); a **`str`** type + bare `HandoffRequest` builds a `ToolOutputSchema` (`tool` mode) but with the `str` text path still allowed, so `tool_choice` stays `auto`, not `required`. Either way handoff is **thinking-safe on every provider** — only a `ToolOutput` forces `allow_text_output=False`. Verified by building the schemas against pydantic-ai 1.47.0 (vendored) and 1.107.0 (latest). The output-tool design is abandoned; there is no Anthropic-thinking guard to write.
- **Per-turn directory ⇒ a per-run model via `create_model` (memoized).** The Handoff-scoped live directory changes per turn, so `HandoffRequest` is built per run and put in the per-run `output_type` override (which **replaces**, carrying the full `[final_output_type, HandoffRequest, DeferredToolRequests]` — dropping `DeferredToolRequests` breaks tool dispatch; the per-run override is sound because calfkit registers no output validators against the construction-time `output_type`):
  - `__base__=HandoffRequest` — a **stable base class**, so `isinstance` discrimination holds across per-turn rebuilds;
  - `__doc__` = the rendered directory (same render code as messaging; handoff-flavored *"hand off control; you will not regain it"* + `name — description` per live agent; self-excluded; empty → no agents) — the structured-output analog of a tool docstring, surfaced to the model as the option's description (native schema / prompted instructions);
  - `name: Literal[<live agent names>]` — the provider's structured-output validation **rejects an out-of-directory target natively**, and pydantic-ai **auto-retries** an invalid value (`RetryPromptPart`; `_output.py`→`ToolRetryError`→`_agent_graph.py` `increment_retries`) with **no calfkit code**; self is excluded from the directory, so self-handoff is rejected the same way;
  - **memoized** by the live-directory signature (`@lru_cache` on the sorted `(name, description)` tuple) → the model + its compiled schema are built **once per distinct live set**, not every turn.
  - **empty live set ⇒ omit the member + an ephemeral instruction.** When a `Handoff` handle is present but **no** in-scope peer is live that turn, `HandoffRequest` is **omitted** from `output_type` — the per-turn builder returns the base `[final_output_type, DeferredToolRequests]` (≡ the construction-time type, so no override is applied). A `Literal` has no empty form (verified: building a pydantic schema over an empty `Literal` raises `AssertionError: literal "expected" cannot be empty` — `TypeAdapter(Literal[()])` / a `create_model` field; note the type expression `Literal[()]` itself constructs fine, the error is at schema-build; `typing.Never` has no pydantic-core schema; a failing-validator stand-in still *surfaces* to the model as a callable option and faults on retry-exhaustion) — so a "never-valid" placeholder is the wrong tool: never offer the model an action it can only fail. To keep the **dormant capability legible**, an **ephemeral instruction** is **concatenated with** any caller `temp_instructions` into the per-run `instructions=` of the model call — passed as a list with `None`/empty filtered (per-run instructions append to the construction-time system prompt; do **not** substitute for `temp_instructions`, and never pass a bare `None` element — pydantic-ai routes a non-`str` element to a callable and would raise): *"You cannot currently hand off this conversation or task to another agent, as no other agents are online."* Per-run instructions are request-time text on the `ModelRequest.instructions` field — **not** a conversational part (verified), recomputed each run, invisible to POV projection (`_projection.py` reads parts, never `instructions`) — so this rides the same ephemeral channel as the system prompt and tool/output-schema descriptions, and **self-heals** the moment a peer comes online (the `Literal` member returns, the note disappears). Generic, so uniform across curated and `discover` scope; injected **only** when a `Handoff` handle is present **and** the live in-scope set is empty (a routine operational state — curated peers all offline; `discover` with only-self / pre-advert startup / a degraded view, §9). (Messaging needs no analog: its tool is always injected, so an empty directory renders as a "none reachable" tool body and the capability stays visible — §5.2.)
- **The handoff turn (C2).** `message` is the handing agent's **summary/context for the peer** (not merely a reason — work may be done), **required non-empty** — enforced by `Field(min_length=1)` on the base `HandoffRequest.message`, so an empty value is auto-retried by pydantic-ai just like an out-of-`Literal` `name`. (Do **not** rely on projection's empty-surface omission to catch it — a populated object with `message=""` still surfaces.) A's `HandoffRequest` output is **persisted to `message_history` like any structured output** — no special turn, no synthesized note. When B continues, POV projection renders A's output **cross-agent as a JSON string attributed to A** — `<A>\n<json>` (`_surface` + `_render_structured_args`). The JSON shape is **mode-dependent** but always legible: in *tool* mode it's the flat args `{"message": …, "name": …}`; in *native/prompted* mode it's the model's union-wrapped text `{"result":{"kind":"HandoffRequest","data":{…}}}`. Either way B reads the handoff (its `message`, and that it came from A via the `<A>` prefix) from A's **actual** output — truthful and correctly attributed; no prose is fabricated or mis-credited to A. **Cross-agent rendering (structured-output agents):** for a **structured** `final_output_type`, the union renames the output tool `final_result_HandoffRequest`; the `_surface` `final_result`-*prefix* match (**landed in #276** — `_is_output_tool`, a pure handoff-unaware renderer that also closed the pre-existing C1b for any union-output agent), on top of the viewer-aware POV fix (#268), renders it. For the common **`str`-output** agent the union's lone output tool is named exactly `final_result`, so its handoff surfaced even pre-#276; native/prompted mode renders A's output as a verbatim JSON `TextPart` regardless. (Pre-#276 a structured agent's handoff was **silently dropped** for the peer — which is exactly what #276 fixed.)
- **Transfer (§5.4) → `TailCall`** carrying the **current** conversation State: the framework pops-and-re-pushes the same frame retargeted to `derive_input_topic(name)`, preserving `frame_id`/`tag`/**`callback_topic`** and the `caller_node_id` (`base.py:573-580`). So B inherits the original caller's return address + full conversation, continues, and returns to **A's original caller**; A drops out. **The caller's per-run overrides are NOT carried to B.** (These are the *caller's* per-invocation `tool_overrides`/`model_settings`, set on the client call — not "A's"; B is a different agent and uses its own tools/model.) Clearing requires nulling **both** override channels, because they are independent: the agent reads `state.overrides`, but `prepare_context` re-applies `frame.overrides` onto `state.overrides` at B's start (`base.py:421-422`), so nulling the state alone is silently re-overwritten (M1). **Mechanism:** the handoff nulls `state.overrides` on the carried State (`run()` owns `ctx.state`) **and** sets a new opt-in **`clear_overrides=True`** flag on the handoff `TailCall`; `_publish_action`'s TailCall arm honors it via `replace(frame, …, overrides=None)` (default `False` → the generic `TailCall` keeps **preserving** `frame.overrides`, which is correct for the self-retry to *self*, so clearing is handoff-only). Verified non-breaking: the only existing `TailCall` (the all-invalid self-retry) and all test TailCalls default to `False`, and `TailCall` is an in-process `NodeResult` action (not a wire model), so the publish arm is a true no-op for them. This clearing is thus **specific to a genuine handoff** (target = another agent); the staleness self-retry below TailCalls to *self* and keeps the caller's surface. The transient tool state is clean (handoff is A's terminal output). Carrying the full conversation is correct (B *becomes* the agent) and adds no new injection surface (§7).
- **Bad target — two layers, the second a thin modular guard.** (1) An invalid name (hallucinated / out-of-scope / self) is impossible-or-auto-retried by the `Literal` + pydantic-ai output-retry — **no calfkit handling**; exhausting the per-run retries surfaces as pydantic-ai's `UnexpectedModelBehavior` (general output-retry behavior, rides the fault rail). (2) The render→dispatch **staleness race** (a name live at render that went offline before dispatch) is the *only* case calfkit handles: a **thin, self-contained liveness re-check**, built in the **final-output branch** (where `result.output` is the `HandoffRequest` — the non-`DeferredToolRequests` disposition; this *mirrors the pattern* of the existing all-invalid self-retry, which lives in a different code path, the deferred-tools branch). It re-reads the live view and, if the target is gone, does **not** relinquish — it **self-retries** (appends a feedback turn + `TailCall`s to self, which re-renders a fresh `Literal` minus the offline agent). The guard is a **drop-in** — encapsulated so it does not leak into the surrounding dispatch code. **Bound:** this self-retry is **not bounded** today — pydantic-ai's retry budget resets on each Kafka re-entry, and there is no calfkit-level counter that survives re-entry, so a pathological re-render→still-stale loop is **unbounded in v1** (the same class as an unbounded handoff ring, §8). A re-entry-surviving hop/retry counter is the **#251 backstop** (§14). The staleness window is sub-second so the loop is unlikely, but it is *not* bounded by construction.
- **Composition:** chained handoffs work (each re-targets the one frame; the last agent returns to the original caller). A handoff *inside* a peer-message is transparent — if A `message_agent`s B and B `handoff`s to C, C's answer folds into A's tool result.

### 5.4 Dispatch resolution

On the model's output, the agent routes by **disposition** — tool calls and the final output are separate code paths. **Tool calls** are registered (`add_tool_call` — load-bearing for the completion check, which `KeyError`s on an unregistered id) and dispatched (first two routes below); the **final output** (`result.output`) is dispatched on its own branch (third):
- **External tool** → `Call(tools_registry[name].dispatch_topic, ctx.state, body=ToolCallRef, tag)`.
- **`message_agent`** → resolve+validate target (view re-read at dispatch; cycle check §8) → `Call(derive_input_topic(name), fresh_seeded_state, tag, isolate_state=True)` or `RetryPromptPart`.
- **`HandoffRequest`** — handled in the **final-output branch** (not a tool-call route; `isinstance(result.output, HandoffRequest)`): a thin liveness re-check (§5.3) → live: persist A's output normally + `TailCall(derive_input_topic(name), state=ctx.state, overrides cleared)`; stale: self-retry. Invalid names never reach here — the per-turn `Literal` + pydantic-ai auto-retry handle them.

**Parallel fan-out + the lone-`message_agent` case (per-kind — the converged a2a decision).** When the model emits multiple calls in a turn, the `Call` list is built **per-kind**: `message_agent` siblings get `(derived input topic, fresh seeded sub-state, no ToolCallRef body)`, tool siblings get `(static dispatch topic, deep-copied caller state, ToolCallRef)`; they open one durable fan-out batch and every reply folds by `tag` at closure. A sibling that fails validation becomes an immediate `RetryPromptPart` and is **excluded from the dispatched batch**, so the dispatched-slot set still matches the completion check. **A lone `message_agent` (N=1) does NOT take the single-`Call` fast path** — it opens a **degenerate one-element durable batch** so the caller's state is snapshotted at dispatch and restored at fold (C1): the fast path adopts the *callee's* returned state, which is correct only for a tool (it round-trips the caller's own state and never authors `message_history`), but a peer runs on a fresh seed and authors its own sub-conversation, so without the snapshot the caller would resume on the peer's history. N=1 and N≥2 messaging thus share one snapshot/restore path; a lone *tool* call keeps the fast path.

**Impl note (the OPEN-trigger signal).** The durable store today forbids a singleton batch — `FanoutOpen.expected` is `min_length=2` (docstring: a batch-of-one is "a stateless continuation, never registered") and the OPEN trigger (`base.py`) keys purely on `len(output) >= 2`; a bare `Call` *and* a 1-element `list[Call]` both take a no-snapshot path (the latter is a documented "stateless continuation"). Enabling the lone-`message_agent` batch is **two changes** (Option A, locked):
- (a) relax `FanoutOpen.expected` to `min_length=1` and rewrite **both** its docstring **and** the `base.py` OPEN-trigger comment that today calls a 1-element list "a stateless continuation, never registered" — that semantic holds only for an *unflagged* 1-element list now, so the comment must be re-qualified (not just the number; loosening a validator is backward-compatible);
- (b) add an explicit **`isolate_state: bool = False`** field to the `Call` action (`models/actions.py`) — an in-process `NodeResult` action, **not** a wire model, so additive/non-breaking (mirroring the `clear_overrides` `TailCall` flag, §5.3). The agent sets `isolate_state=True` on every `message_agent` `Call`; the OPEN trigger generalizes from a count to the **semantic property** — open a durable batch when `len(calls) >= 2` **or** `any(c.isolate_state for c in calls)` (normalizing a bare `Call` to a one-element list first). A lone *unflagged* `Call`/1-element list keeps its existing no-snapshot path; only an `isolate_state` call forces the degenerate batch. **The normalization must happen at the dispatch disposition site, *before* the fan-out check** — the current OPEN trigger matches only `isinstance(output, list)`, so a flagged *bare* `Call` must be coerced to `[call]` first; otherwise it falls through to `_publish_action`'s bare-`Call` fast path (`base.py` ~:527) and skips the snapshot, silently reintroducing C1. Make that branch provably unreachable for an `isolate_state` call. An **all-excluded** turn (every dispatched call fails validation — e.g. a lone `message_agent` with a bad target) never reaches OPEN: it lands `RetryPromptPart`s and returns the pre-existing all-invalid self-retry `TailCall` (`agent.py` ~:474-475) *before* the fan-out build, so OPEN only ever sees ≥1 dispatched call — `min_length=1` is never confronted with a 0-slot batch.

base.py stays **generic** — it reads a property of the call (*"the callee runs on a state isolated from the caller's; snapshot/restore so the caller resumes on its own state, not the callee's return"*), never `message_agent`-specific logic. The field names the canonical pattern (a peer message is the *same* control flow as a tool `Call` — invoke→await→fold — differing only in state isolation, so an attribute, not a new action type). The store's invariant generalizes from "true fan-out (N≥2)" to **"register a batch iff the caller's state must survive the call independently of the round-trip"**; the fold/close/restore machinery already handles N=1 (the completion gates are set-equality), so this is a field + guard/trigger relaxation, not new fold logic. (A `HandoffRequest` is the turn's *output* — not a `Call` — so it never rides a fan-out batch.)

## 6. Worker wiring

- **Write side (every agent):** each hosted agent contributes the `AgentCard` advert; the worker stands up the `calf.agents` publisher (boot dependency for any agent-hosting worker, §11).
- **Read side (gated, distinct key):** a `ControlPlaneView[AgentCard]` over `calf.agents`, under a **new `AGENTS_VIEW_RESOURCE_KEY`** (≠ `CAPABILITY_VIEW_RESOURCE_KEY`), via a **new `_maybe_register_agents_view`** gating on **any `peers` handle** (Messaging or Handoff) on a hosted agent.

## 7. Discovery scope & trust (v1)

- **Curated** (named handles) — directory + validation restricted to the named live peers; the **only** v1 trust boundary, caller-side, per capability.
- **Open** (`discover=True`) — all live agents, for that capability.
- **No callee-side control.** Every agent is advertised *and* reachable (anyone knowing a name can derive its input topic); a peer can be messaged *or* handed control without opting in. Visibility + admission are deferred whole to the ACL plane (#231). A messaged peer treats the inbound `message` as untrusted (§5.2); a handoff adds no *new* injection surface (§5.3) — but it **propagates** whatever the carried conversation already holds, including an untrusted `message_agent` seed planted at an earlier hop, so the §5.2 "guard the seed" obligation rides along to the handoff target.

## 8. Recursion & loop bounds

- **Messaging cycle guard (ADR-0016).** `CallFrame` gains `caller_node_id`/`caller_node_kind` — the *dispatching* node, stamped **only at the genuine `invoke_frame` push** and **preserved verbatim** by `replace()` in `TailCall`/`mark_fanout` (never re-stamped to self). `prepare_context` derives a read-only `ctx.ancestor_callers` from the inbound frames; the dispatch resolver rejects a `message_agent` whose target `(name, "agent")` is already an ancestor → `RetryPromptPart`. Matching by *identity* (not topic) catches public-entry cycles; the per-branch inbound stack allows a legitimate diamond but blocks a ring. **Cycle detection alone guarantees termination** for messaging in a finite mesh.
- **Handoff loops are unbounded in v1 — and this guard cannot see them.** `TailCall` *replaces* the frame, so the ancestor chain never grows; an A→B→A handoff ring has nothing for the identity check to match. Per design, **no guard in v1** — a documented operational hazard owned by the model/developer, with a handoff-hop counter as the obvious #251 backstop.
- **Deferred to #251:** depth/hop/cost/breadth budgets for both.

## 9. Liveness & failure edges

- **Stale at render** — live-only directory; a stale peer is never listed.
- **Stale between render and dispatch** — re-checked at dispatch → `RetryPromptPart` (messaging) or a thin liveness re-check → self-retry (handoff, §5.3). For handoff, invalid names are handled earlier by the `Literal` + pydantic-ai auto-retry; only this staleness race reaches calfkit.
- **Handoff with an empty live directory** — a `Handoff` handle present with no in-scope peer live: the `HandoffRequest` member is **omitted** (an empty `Literal` is unbuildable) and an ephemeral instruction conveys the dormant capability (§5.3). A routine, self-healing operational state (curated peers all offline; `discover` with only-self / pre-advert startup / a degraded view), **not** an error.
- **Degraded/failed agents view** — warn-and-degrade (mirror the capability view); never raise on reader health.
- **Replicas** — the name-scoped input topic is a consumer group; a survivor handles the message/handoff; the collapsed view yields one card per name.
- **Mid-message no-reply** (a messaged peer dies before replying) — under at-most-once (`ACK_FIRST`) the calling slot dangles, with no per-call deadline; it leaks the durable batch (no batch-level deadline) — in a *mixed* fan-out batch it also strands the completed tool siblings, and since a lone `message_agent` now rides a **degenerate one-element batch** (C1/§5.4), even a single peer message leaks that one-slot batch on peer death. The SDK's general no-reply gap, owned by #251. **Documented hazard, not benign.** (Handoff has no reply to A by construction, so this is a messaging-only edge.)
- **Handoff co-emitted with a tool call (native/prompted mode)** — when the resolved output mode is native/prompted (e.g. Anthropic + extended thinking), the `HandoffRequest` is JSON *text*; if the model emits a function/external tool call in the *same* response, pydantic-ai processes the tool call and **defers** the handoff text to a later turn (it prioritizes tool calls over text). In tool mode, `end_strategy='early'` instead drops the sibling. The handoff is delayed, never corrupted — benign, but documented (§5.3, M3).
